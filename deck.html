The Solution

Hardware-Isolated Verification

Exploits are encrypted to an AWS Nitro Enclave — a hardware-isolated VM with no network, no storage, no shell access.


The enclave verifies the exploit works, creates a marketplace listing, and re-encrypts for the buyer. The platform operator never sees the plaintext.


  • Seller encrypts PoC client-side (ECIES P-384)
  • Database stores only ciphertext
  • Enclave decrypts, runs, destroys on termination
  • Only pass/fail leaves the enclave
# Seller's perspective: $ ndai-seal attest https://zdayzk.com PCR0: 97663b18...f9b2f7 Mode: nitro (AWS hardware) $ ndai-seal verify-pcr0 --pcr0 97663b18... MATCH — on-chain registry ✓ $ ndai-seal encrypt \ --poc exploit.sh \ --key enclave.der Sealed: 196 bytes → sealed.bin Only the enclave can decrypt.

Capability Oracle

The Seller Doesn't Define Success

The enclave plants random canary values in the target filesystem. The seller's PoC must retrieve them to prove capability.

# Inside the enclave: canary = os.urandom(16).hex() # random # ACE canary (world-readable) write("/var/lib/ndai-oracle/ace_canary", canary) # LPE canary (root-only) write("/var/lib/ndai-oracle/lpe_canary", canary) chmod(0o600) # Run PoC → check stdout for canary if canary in poc_stdout: capability = VERIFIED
